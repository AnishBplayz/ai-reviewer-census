## Why there is no per-vendor action rate here

Action rates **are** computed per vendor and stored in `data/report.json`. They are deliberately not rendered as a comparison table, because the data cannot yet support the ranking readers would take from one:

- Reviewers differ in volume. A tool posting more threads per PR mechanically scores a lower hit rate, independent of quality.
- Vendors occupy largely non-overlapping sets of repositories, so their scores reflect different codebases, languages, and review cultures.
- The `isOutdated` proxy is sensitive to *when* in a pull request a comment lands, and reviewers differ systematically in that.

None of these are controlled for. The aggregate human-versus-AI gap above is reported because it is large, stable across sample sizes, and biased *against* the conclusion drawn from it. A vendor-versus-vendor gap is neither.

The raw scans are public. If you want to compute it, `data/scans.jsonl` has everything.

## Method and limitations

- **Sample frame:** public, non-archived repos above a star floor with a recent push, taken from GitHub search ordered by last update. Biased toward popular open source and away from private corporate monorepos, where multi-bot adoption is plausibly higher. The headline number is therefore best read as a **lower bound**.
- **Per-repo depth:** the most recently updated PRs only, so a repo that adopted a second reviewer years ago and dropped it will not register.
- **Action rate** relies on GitHub's `isOutdated` flag, described above.
- **Bot identification** is an explicit allow-list of known logins. New or self-hosted reviewers under custom app names are undercounted.
- No LLM is used anywhere in this analysis. Every number is reproducible from the raw scans in `data/scans.jsonl`.
